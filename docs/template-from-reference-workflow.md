# Template From Reference Workflow

这份文档记录一种 **从参考图新增模板** 的稳定流程。

适用场景：
- 用户发来一张成品图 / 截图 / 风格参考图
- 目标不是像素级还原，而是抽象出一个可复用的模板
- 后续会继续用 JSON 数据批量产图

这次 `apple-notes-handwrite` 模板就是按这条流程做出来的。

---

## 目标

把一张参考图拆成：
- 可复用的布局结构
- 可参数化的数据字段
- 可维护的模板代码
- 可回归验证的示例图

最终交付应该包括：
- 模板源码
- 模板注册
- 示例 JSON
- README showcase
- 可直接渲染的输出图

---

## 一、先理解参考图，不急着写代码

收到参考图后，先做 **结构识别**，不要一上来就抄颜色和字号。

优先拆这几层：

### 1. 画布层

先判断：
- 画布比例（常见是 1080x1440 / 3:4）
- 背景颜色或背景材质
- 是否有边距、安全区、留白节奏

### 2. 主结构层

把图看成几个大块：
- 顶栏
- 标题区
- 正文区
- 装饰区
- 页脚 / 底部信息

这一步只关心布局，不关心精细视觉。

### 3. 装饰元素层

判断哪些是：
- 固定装饰（如线条、分隔线、卡片阴影、圆角）
- 系统 UI 风格元素（如 iPhone Notes 的返回、分享、更多）
- 内容无关元素（如水印、贴纸、截图状态栏）

内容无关元素一般不要进模板。

### 4. 文本风格层

观察：
- 字体风格（黑体 / 圆体 / 手写 / marker / 书法）
- 标题是几行
- 字号大概占了多少空间
- 字间距偏紧还是偏松
- 行高密还是松

### 5. 参数识别层

最后再决定哪些应该开放成输入字段，比如：
- `title`
- `subtitle`
- `bullets`
- `label`
- `icon`
- `day`
- `theme`

原则：
- 先少后多
- 只抽取真正会重复变化的字段
- 不要为了“灵活”把所有东西都参数化

---

## 二、定义模板边界：什么做，什么不做

新增模板前先明确三件事：

### 1. 需要保留的风格

例如：
- Apple Notes 风格顶栏
- 横线纸背景
- 中文手写感标题

### 2. 可以近似的部分

例如：
- icon 可以用矢量近似，不必截图抠图
- 字体不要求完全同款，但风格必须接近
- 手写感可以通过字体 + 微旋转 + 行距来模拟

### 3. 不进入模板的部分

例如：
- 右下角水印
- 截图里临时性的系统状态
- 用户内容之外的营销文案

如果用户明确说某部分忽略，直接在实现里排除。

---

## 三、实现步骤（代码层）

## 1. 新建模板文件

位置：

```txt
src/templates/<template-name>.tsx
```

模板文件需要包含：
- `name`
- `description`
- `defaultWidth`
- `defaultHeight`
- `examplePath`
- `schema`
- `render(input)`

建议先把大结构搭起来，再补细节。

---

## 2. 注册模板

更新：
- `src/types.ts`
- `src/templates/registry.tsx`

至少要做：
- 把模板名加入 `BUILTIN_TEMPLATES`
- 在 `registry.tsx` 里 import 并注册

如果漏掉这步，CLI 不会识别新模板。

---

## 3. 增加标题排版 profile

如果模板有大标题，必须更新：

```txt
src/templates/title-engine.ts
```

这里要定义：
- 标题容器宽高 `box`
- 初始字号 `fontSize`
- 最小字号 `minFontSize`
- 行高 `lineHeight`
- 字距 `letterSpacing`
- 最大行数 `maxLines`
- 偏好行数 `preferredLines`

经验：
- 先让标题“看起来占得够满”
- 再微调不要太撑
- 手写风格一般字距不要太紧，行高也不要太死

---

## 4. 增加 fixture 支持

如果项目里要跑标题回归，更新：

```txt
src/fixtures.ts
```

给新模板补一条 `buildFixtureInput()` 分支。

这样后面批量验证长标题、短标题、换行稳定性会轻松很多。

---

## 5. 增加 example JSON

位置：

```txt
examples/<template-name>.json
```

要求：
- 用最能代表模板风格的一组内容
- 能直接渲染
- 尽量覆盖这个模板的关键输入字段

这个 example 既是 demo，也是后续调样式时的基准样本。

---

## 四、图标与字体的处理原则

## 1. 图标

如果参考图里有系统 icon：
- 优先自己画简单矢量
- 不要直接用奇怪的字符替代
- 不要依赖截图裁切图标

推荐做法：
- 用 `svg` 画返回箭头、分享、更多这类 icon
- 控制线宽、圆角、间距，尽量贴近原系统风格

经验：
- `‹` 这种字符临时看着像，但很容易不够精致
- 真正稳定的是自定义 `svg path`

## 2. 字体

从参考图判断字体时，先分类型：
- 系统正文风格
- 中文手写风格
- marker 风格
- 书法风格

如果参考图是中文内容，**不要只找英文手写字体**。

优先找：
- 支持中文
- 可商用 / 开源
- 风格接近目标图
- 本地可稳定渲染

这次实践里的结论：
- `Patrick Hand`：英文手写感还行，但不适合中文模板主方案
- `LXGW Marker Gothic`：中文 marker 手写感更强，适合标题
- `LXGW WenKai Lite`：中文手写正文感更自然，适合更柔和的风格

原则：
- 不追求“完全同字体”
- 追求“整体风格成立”

---

## 五、迭代方式：一次只改 2-3 个变量

从参考图到最终可用版本，通常不会一次到位。

建议每轮只改：
- icon 风格
- 字号 / 字距 / 行高
- 字体
- 顶部位置 / 左右边距

不要一轮里把所有参数全改掉，不然很难判断到底哪个改动起作用。

典型顺序：
1. 先让结构对
2. 再让标题占位对
3. 再修 icon
4. 最后调字体气质

## 六、输出与验收流程

## 1. 本地渲染

先在 repo 内验证能否正常出图：

```bash
node dist/cli.js render \
  --template apple-notes-handwrite \
  --data examples/apple-notes-handwrite.json \
  --out out/apple-notes-handwrite.png \
  --font <font-path> \
  --fontBold <font-path>
```

## 2. 输出到约定目录

如果用户有固定查看目录，直接输出到那里。

这次约定目录是：

```txt
/Users/fox/Documents/zspace-cache/ZSPACE/sata1-18611681815c/download/xiaohongshu/postgen-shots/
```

原则：
- 让用户能直接查看
- 不要让用户自己在 repo 和 NAS 之间找文件

## 3. 用户反馈后二次调优

用户通常会指出：
- icon 不像
- 字太大 / 太小
- 字体味道不对
- 间距不对

收到反馈后，继续基于同一张图做迭代，不要重新发明结构。

## 七、README / Showcase 更新

模板确认可用后，补两处：

### 1. `README.md`

在 showcase 中增加模板项：
- 标题
- 对应截图

### 2. `website/shots/`

把最终确认版图片复制进去：

```txt
website/shots/<template-name>.png
```

注意：
- README 展示图应该使用最终确认版，而不是中间试验版
- 最终版和 NAS 里的验收图最好保持一致

---

## 八、提交代码前的 checklist

新增参考图模板后，提交前至少检查：

- [ ] `src/templates/<name>.tsx` 已新增
- [ ] `src/templates/registry.tsx` 已注册
- [ ] `src/types.ts` 已加入模板名
- [ ] `src/templates/title-engine.ts` 已补 profile
- [ ] `src/fixtures.ts` 已补分支（如需要）
- [ ] `examples/<name>.json` 已新增
- [ ] `website/shots/<name>.png` 已更新
- [ ] `README.md` 已加入 showcase
- [ ] `npm run build` 通过
- [ ] 能实际渲染出图

## 九、建议的协作方式（给后续用）

以后如果用户继续通过“发一张图片”来新增模板，推荐按下面协作：

### 用户提供
- 一张参考图
- 哪些元素要保留
- 哪些元素忽略
- 输出目录（如果有固定要求）

### 助手执行
1. 先理解参考图布局与风格
2. 明确哪些元素进模板、哪些忽略
3. 新增模板并注册
4. 选一个风格合适的字体方案
5. 出首版图给用户看
6. 根据反馈迭代 icon / 字体 / 间距 / 布局
7. 确认后更新 README showcase
8. commit + push

## 十、这次 `apple-notes-handwrite` 的经验总结

这次实践里几个关键经验：

### 1. 先判断“中文手写感”，不要误判成英文字体问题

参考图虽然看起来像手写便签，但核心是：
- 中文内容
- 中文手写风格
- Notes 风格 UI

所以字体选择必须优先中文可用。

### 2. icon 不能偷懒用字符替代

临时用字符模拟：
- 上手快
- 但最终质感差

最终还是应该上 `svg`。

### 3. 字体只解决一半问题

即使换对了字体，最终像不像还取决于：
- 字号
- 字距
- 行高
- 左边距
- 微旋转

### 4. README 展示图要跟最终确认版同步

不要把中间版本留在 showcase。

## 附：推荐命名规则

建议模板命名：
- 简洁
- 小写
- kebab-case
- 带风格语义

例如：
- `apple-notes-handwrite`
- `xhs-quote-blue`
- `xhs-note-green`

避免：
- 过长
- 带版本号
- 带临时实验后缀

版本应该放在输出文件名里，不要放进模板名里。
